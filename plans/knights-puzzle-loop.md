## North star

A cozy, account-free, single-player web puzzle game where each level is a
partial checkerboard with a start and end square, and you win by moving a
knight so it visits every available square exactly once and finishes on the
end square — running on a local dev port. Puzzles are generated by random
non-revisiting knight walks, so they are **always solvable by construction
(no solver)**. Title: **Knight's Puzzle**.

What must not be diluted: client-side only · valid-by-construction (never add
a solver) · win = cover-all-available-cells **and** end on the end square ·
cutesy but functionally clear, with _which cells are available_ as the
dominant visual.

## SLICE-5a PICKUP — Engine (authored now)

- **Baseline commit:** `3a09058` (scaffold, all gates green).
- **Goal:** a pure, seeded engine module that the UI and tests both import.
  No DOM, no React — must typecheck under both tsconfigs.
- **Proposed shape** (reviewer may refine):
  - `type Cell = { r: number; c: number }`
  - `type Puzzle = { n; available: boolean[][]; start: Cell; end: Cell;
path: Cell[]; seed: number }`
  - `knightMoves(cell, n): Cell[]` — in-board knight destinations.
  - `makeRng(seed): () => number` — deterministic PRNG (e.g. mulberry32).
  - `generatePuzzle(n, k, seed): Puzzle` — random non-revisiting walk of up
    to `k` steps from a random start; visited cells become `available`;
    `start = path[0]`, `end = path[last]`. Retry/extend to hit a usable
    length; clamp `k <= n*n - 1`; guarantee `path.length >= 2`.
  - `legalMoves(puzzle, current, visited): Cell[]` — knight moves that are
    available and not yet visited.
  - `isWin(puzzle, visited: Cell[]): boolean` — visited covers exactly all
    available cells once, each step a knight move, starts at start, ends at
    end.
- **Load-bearing mechanics / traps:**
  - **Determinism:** same `(n, k, seed)` MUST produce the same puzzle —
    tests depend on it. No `Math.random()` in the engine; thread the RNG.
  - **Dead ends:** a walk can get stuck before `k` steps (all knight
    neighbors visited). That's fine — accept the shorter path; just ensure
    length ≥ 2 and, for a target difficulty, retry a few seeds/starts to
    reach a minimum length. Document the retry policy.
  - **Coordinates:** pick ONE convention (row, col) and keep it everywhere;
    knight deltas are the 8 (±1,±2)/(±2,±1) combos.
- **Acceptance criteria:**
  - Unit tests cover: knightMoves (corner=2, center=8, edge counts);
    generator produces a valid knight path with no revisits, available count
    == path length, correct start/end; determinism (same seed → deep-equal
    puzzle); isWin true for the generator's own path, false for partial /
    wrong-end / non-knight-step; legalMoves filters correctly.
  - `bun run ci && bun run smoke` all green (smoke still passes on skeleton).
- **Decide-with-reviewer:** the `Puzzle` shape and the generator's
  retry/length policy; difficulty-number ranges can wait for 5d.
- **Locked (don't relitigate):** no solver; determinism via seeded RNG;
  engine stays pure (no DOM/React).

## SLICE-5b PICKUP — Minimal playable (authored after 5a)

- **What 5a taught (fold in):**
  - The two-tsconfig split holds; the engine is pure and imported by both UI
    and tests. Generator signature is `generatePuzzle(n, steps, seed)` — the
    middle arg is **steps** (knight moves), max cells = `steps + 1`.
  - `Puzzle.path` is the witness solution — use it as the smoke's solve route.
  - Reviewer expects: documented constants, explicit failures (no silent bad
    states), no mutation of caller collections, and non-flaky tests.

- **Goal:** a minimal but real playable game on a dev port. Generate a puzzle,
  knight on start, click a legal cell to move, track visited, detect win.
  `window.__KP__` exposes the full authoritative state. Ugly is fine (cute
  comes in 5c).

- **Load-bearing mechanics / traps:**
  - **Evidence surface:** `window.__KP__` must expose `knight`, `visited`,
    `visitedCount`, `totalCells`, `legalMoves`, `won`, and the witness
    `solution` (= `puzzle.path`) so the smoke can solve the REAL UI via clicks
    and judge by state, never pixels.
  - **Smoke contract:** every available cell carries a stable
    `data-cell="r-c"` attribute. After each click, poll `__KP__.knight` until
    it matches (click-and-verify — clicks race re-renders).
  - **Game logic:** keep transitions in a pure `src/game.ts`
    (`newGame`/`tryMove`/`currentLegalMoves`) so they unit-test without a DOM.
    `tryMove` no-ops on an illegal click and recomputes `won` via `isWin`.
  - **Seeds:** initial seed is random (`Date.now`/`Math.random` OK in app code,
    NEVER in the engine). The smoke reads `solution` from `__KP__`, so it is
    seed-agnostic.
  - **Stuck states:** a dead end before covering all cells is possible; 5b
    ships a "New puzzle" button to recover (undo is 5c).

- **Acceptance criteria:**
  - Renders a clearly-partial board (start n=6, steps=12), knight on start,
    end marked; clicking a legal cell moves the knight; illegal click is a
    no-op; covering all cells ending on the end square shows a win state;
    "New puzzle" regenerates.
  - Unit tests for `src/game.ts`: newGame placement; illegal move no-op;
    legal move advances + records; following `puzzle.path` wins and then
    `currentLegalMoves` is empty.
  - Smoke gate rewritten to SOLVE the real UI via clicks and assert
    `__KP__.won === true` and `visitedCount === totalCells`.
  - All gates green (`bun run ci && bun run smoke`).

- **Decide-with-reviewer:** the `window.__KP__` shape (esp. exposing the
  witness `solution`), the initial params (n=6, steps=12), and the
  `data-cell` selector contract.

- **Locked (don't relitigate):** no solver; win = cover-all-cells AND end on
  end; judge via the evidence surface; client-side only.

- **Resources:** `src/engine` (exports), `src/game.ts` (new), `src/App.tsx`
